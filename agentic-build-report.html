Overview

Two tiers, one contract

Aegis ID is built by a two-tier agent system. A frontier orchestrator — Claude, running in Claude Code — owns the thinking: specifications, failing tests, security review, and merges. Local implementer agents, powered by an open-weight model served on-device by oMLX, write the bulk of the code.

Between the two tiers sits a layer of deterministic quality gates that costs nothing to run and cannot be persuaded. The economics follow directly: frontier tokens are spent only where judgment matters, iteration churn runs locally for free, and the platform's core security invariant — deny by default — applies to the agents themselves, not just to users.

Operating Model

The organization chart

If the agents were a team, this is how they would report. The product owner sets direction; the orchestrator acts as engineering manager and security officer — it never writes what it can delegate, but signs off on everything. Four developers own their areas of the codebase. QA is an automated department nobody can skip, and Compliance reports to management alone.

Reporting lines are enforced by scripts, not prompts. Dotted lines: shared infrastructure use, and Compliance's direct escalation channel to management.

The dotted lines matter. The coders use the model server but do not own it, and Compliance never negotiates with developers — a violation goes straight up the chain, and the work comes back down as an orchestrator task.

Model Selection

The models, and why each one

Claude Fable 5 — the orchestrator

Anthropic's frontier model, accessed through Claude Code. It holds the roles where a mistake is expensive and judgment is irreducible: translating intent into precise specifications, authoring the failing tests that act as contracts, and reviewing every diff against the platform's security invariants. For a decentralized-identity product aimed at government and banks, the OIDC, WebAuthn, and credential-flow review work is held to frontier standard without exception — which is exactly why it is the only tier allowed to touch protected paths.

Devstral Small 2 — the implementer

A ~24B-parameter, Apache-2.0-licensed coding model from Mistral AI, purpose-tuned for agentic software engineering — long-horizon tool use, repository navigation, and edit-test loops — rather than general chat. At 4-bit MLX quantization it occupies roughly 13 GB, which is what makes it the largest capable coder that fits the 24 GB machine's 13–17 GB envelope with KV-cache headroom at 32K context. Its job is deliberately narrow: implement well-specified, scoped tasks against a failing test, with a named exemplar file to pattern-match. Small models are unreliable inventors but strong pattern-transfer engines, and the spec discipline plays directly to that strength. Comparable alternatives in the same weight class — a Qwen coder in the 9–14B range if more speed is wanted, or higher-quant smaller models — slot in by changing one variable in delegate.sh.

GLM-5.2 (hosted) — the optional burst tier

The original design target for the coder tier, and still the right ceiling for it — but at 4-bit it wants hundreds of gigabytes of memory, which is Mac Studio territory, not MacBook. Because oMLX, hosted open-model APIs, and Anthropic's API all speak compatible endpoints, a task that exceeds the local model can be escalated to a hosted GLM-5.2 (or equivalent open frontier model) by switching BASE_URL — keeping per-token cost far below frontier pricing while preserving the identical spec-gate-review pipeline. Note the trade-off honestly: burst-tier traffic leaves the device, so it is governed by the same rule as everything else — nothing in SECURITY_PATHS.txt is ever implemented outside the orchestrator.

One model per tier, not per agent. The four implementer roles share a single set of local weights. Four different models would thrash 24 GB of unified memory for no quality gain — role behavior comes from prompts, scoped AGENTS.md guides, and worktree isolation, which cost nothing.
Process

The task loop

The orchestrator sees a spec going out and a verified diff coming back — never the iteration churn in between.

Four rules hold the loop together, and every one of them is enforced mechanically rather than by instruction:

The failing test is the contractThe orchestrator writes it before delegating. Implementers never author their own acceptance criteria.
Implementers cannot finish redA task ends only when the gate prints GATE: PASS — no exceptions, no negotiation.
Security paths escalateAny diff touching a protected path fails the gate and returns to management as an orchestrator task.
One task, one worktree, ≤ 5 filesAnything larger is split before it is delegated. Small scopes are what make small models reliable.
Infrastructure

Workstation configuration — MacBook Pro M4 Pro, 24 GB

Unified memory shapes everything. macOS keeps four to five gigabytes; the remainder is shared between model weights, KV cache, and builds. The practical envelope for the model is 13–17 GB — a 4-bit coder in the 9–24B class at 32K context. Run at most two agent sessions concurrently, and drop to one whenever xcodebuild or Gradle is working.

Install the server and a model

brew tap jundot/omlx
brew install omlx

mkdir -p ~/models
pip install -U "huggingface_hub[cli]"
hf download mlx-community/Devstral-Small-2-24B-Instruct-4bit \
  --local-dir ~/models/devstral-small-2-4bit

Serve, with a hard memory cap

omlx serve \
  --model-dir ~/models \
  --max-process-memory 17GB \
  --paged-ssd-cache-dir ~/.omlx/cache \
  --max-concurrent-requests 2

The SSD-paged KV cache is not garnish: agent loops re-read the same repository files, guides, and specs constantly, and the cache serves that context from disk instead of recomputing it.

Wire up the repository

cd aegis-id
chmod +x scripts/agent/*.sh
$EDITOR scripts/agent/delegate.sh   # set LOCAL_MODEL to your model id
scripts/agent/gate.sh web           # baseline must be green first

Confirm the delegation path end-to-end

ANTHROPIC_BASE_URL=http://localhost:8000 \
ANTHROPIC_API_KEY=local \
claude -p "List the services in src/services and say which handle authorization." \
  --model devstral-small-2-4bit

If that answers sensibly, everything downstream works.

Cloud Variant

The same architecture on Azure AI Foundry

For clients who prefer managed cloud over developer workstations — or who need to scale beyond one machine — the entire architecture transplants onto Microsoft Foundry (the platform formerly branded Azure AI Foundry) without changing its shape. The tiers, the contracts, and the deny-by-default governance stay identical; only where each layer runs changes.

Why this lands well with regulated clients

Three properties matter to the government and financial audience. Residency: serverless deployments are hosted in Microsoft-managed Azure infrastructure with prompts and outputs excluded from provider training, while managed-compute deployments keep model traffic entirely inside the client's subscription and virtual network — deployable in Canadian regions for data-sovereignty requirements. Layered safety: Azure AI Content Safety can screen implementer-model output before it ever reaches the gate, adding a filter the local design doesn't have. Procurement: everything bills through the existing Azure agreement — no new vendor relationships, which is often the difference between a pilot and a program.

Hybrid is legitimate. Foundry Local brings catalog models on-device for offline or privacy-first work, so a team can develop against the MacBook pattern and burst to Foundry endpoints for scale — the delegate script only needs a different BASE_URL and key per tier. The governance layer neither knows nor cares where the tokens came from.

Migration in practice

# 1. Deploy models in the Foundry portal (Claude for tier 0, a coder model for tier 1)
# 2. Point the same scripts at the new endpoints:
export ORCH_BASE_URL="https://<foundry-resource>.services.ai.azure.com/..."
export LOCAL_MODEL_URL="https://<coder-endpoint>.inference.ai.azure.com"
# 3. Run gates in pipelines instead of on-device; branch policy requires them
# 4. Keys to Key Vault · endpoints behind Private Link · Content Safety on tier 1 output
Operations

Running a task, and where things live

From the orchestrator session at the repository root: create a worktree, write the spec with the spec-authoring skill together with a failing test, then delegate to the matching role. When the green diff returns, review it — applying the oidc-security-review checklist to anything near authentication or cryptography — merge, and remove the worktree.
