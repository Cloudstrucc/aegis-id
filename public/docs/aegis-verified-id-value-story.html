Product overview

Aegis ID closes the gap between login and trust.

Conventional authentication tells an application that a user signed in. It usually does not prove whether the user still belongs to the right organization, consented to share specific claims, tapped a hardware key, accepted a role change, or personally approved a high-value decision.

Problem 01 Access decisions are scattered.

Roles live in one platform, credentials in another, approvals in email, and audit history in application logs.

Problem 02 Authentication is not consent.

A login event does not prove a user agreed to share claims or accepted a specific business action.

Problem 03 Identity must travel.

Employees, contractors, departments, partners, and external portals often need portable proof beyond one app.

Executive value Reduce risk in sensitive decisions.

Require stronger proof for approvals, administrator changes, claims sharing, badge access, and contractor onboarding.

Management value See who has access and why.

Give administrators a clean organization view of invited, active, disabled, revoked, and elevated users.

IT value Connect to the stack you already use.

Assurance architecture

One policy layer for many kinds of proof.

Aegis ID does not force every organization into one identity pattern. It can use Microsoft Verified ID for portable credentials, YubiKey/FIDO2 for hardware-backed assurance, and OIDC/SAML for standard application integration.

  • Portable credential proof Use Microsoft Verified ID when a credential needs to be presented across departments, portals, or external organizations.
  • Hardware-backed step-up Use YubiKey 5C NFC and FIDO2/WebAuthn when the moment requires phishing-resistant local proof.
  • Application and badge decisions Use Aegis wallet challenges and policy to record approvals, revocations, access changes, and ID checks.

Authentication gets you in. Aegis proves the moment.

Passwordless login, MFA, Conditional Access, and RBAC are essential. They are still not always enough for portable credentials, explicit claim consent, administrator step-up, badge-style identification, or signed business decisions that need a durable evidence trail.

High-assurance workflows

The customer needs Microsoft-native credential proof, hardware-backed step-up, app policy, and an audit trail.

Credential proves eligibility. YubiKey proves hardware-backed presence. Aegis governs and records the decision.

Aegis ID is the assurance control plane. Verified ID and YubiKey are proof methods. The ledger is the record of what happened next.

  1. Choose the assurance path Use Verified ID for portable credential proof or YubiKey/FIDO2 for hardware-backed step-up.
  2. Prove the user or context Authenticator shares a credential, or the YubiKey signs a WebAuthn challenge with user verification.
  3. Approve an expense The app sends a high-value decision payload to the Aegis wallet challenge and policy service.
  4. Show the ledger Credential proof, hardware step-up, approval, consent, and revocation events become visible to the org.

Implementation notes

How this fits real enterprise environments.

Verified ID can be started by QR code or deep link. A QR code is convenient when the request is displayed on a desktop and scanned by a phone; a deep link is better when the user is already on a mobile device. YubiKey is different: it is not a Verified ID wallet. It is a FIDO2/WebAuthn security key that can be used by Aegis ID for hardware-backed sign-in and step-up.

Microsoft-native Verified ID path

Use when the user needs to present a portable employee, contractor, badge, or eligibility credential in Authenticator.

Hardware-backed YubiKey path

Use when the user needs to tap a YubiKey 5C NFC or USB-C key to satisfy phishing-resistant WebAuthn/FIDO2 step-up.

Standards-based apps OIDC / SAML path

Use when Power Pages, Keycloak, Okta, SaaS apps, or custom web apps need claims and assurance from Aegis ID.
