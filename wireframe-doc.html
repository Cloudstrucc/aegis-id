1. Executive Architecture Decision

Cloudstrucc should implement a dual-track architecture: a Microsoft-native production path and a separate Aries interoperability lab.

Production Track

Use Microsoft Entra ID, YubiKey FIDO2/passkeys, Conditional Access, and Microsoft Entra Verified ID for enterprise-grade issuance and verification.

Interoperability Track

Use ACA-Py, Bifold/Credo wallet, mediator, DID service, and VON Network for Aries, DIDComm, AnonCreds, and wallet interoperability experiments.

Important: VON Network is a development Indy ledger. Do not treat it as a production trust registry or production DID infrastructure.

4. Configure YubiKey with Microsoft Entra ID

  1. Enable Passkey (FIDO2) in Microsoft Entra Authentication Methods policy.
  2. Target a pilot group before tenant-wide rollout.
  3. For high assurance, evaluate attestation enforcement and approved YubiKey AAGUID restrictions.
  4. Have users register their YubiKey through Security Info.
  5. Apply Conditional Access requiring phishing-resistant authentication strength for issuer/verifier apps.
NFC: NFC YubiKeys can be used where OS, browser, and app support FIDO2/passkeys over NFC. Test iPhone, Android, Windows, and USB fallback paths.

5. Configure Microsoft Entra Verified ID

Use Quick setup for demos and Advanced setup for production. Advanced setup is preferred when Cloudstrucc needs explicit Key Vault, lifecycle separation, governance, and environment controls.

{
  "credentialType": "CloudstruccEmployeeCredential",
  "displayName": "Cloudstrucc Employee Credential",
  "claims": {
    "employeeId": "string",
    "displayName": "string",
    "email": "string",
    "department": "string",

}

Recommended issuance pattern: ID token hint. The issuer app authenticates the user with Entra ID/YubiKey and passes trusted claims to the Verified ID issuance request.

6. Issuer Application

The issuer app authenticates the user, confirms phishing-resistant authentication, creates an issuance request, renders QR/deep link, receives callbacks, and persists transaction state.

app.post('/api/issuer/create-offer', requireAuth, async (req, res) => {
  await requirePhishingResistantAuth(req.user);

  const request = await verifiedIdClient.createIssuanceRequest({
    credentialType: 'CloudstruccEmployeeCredential',
    claims: {
      employeeId: req.user.employeeId,
      displayName: req.user.name,
      email: req.user.email,
      assuranceLevel: 'FIDO2_YUBIKEY',
      employmentStatus: 'active'
    },
    callbackUrl: `${PUBLIC_BASE_URL}/api/issuer/callback`,
    state: crypto.randomUUID()
  });

  res.json({ qrCodeUrl: request.url, transactionId: request.id });
});

7. Verifier Application

The verifier app creates a presentation request, specifies accepted issuer DID and credential type, validates callback state, and maps verified claims to an authorization decision.

app.post('/api/verifier/create-request', async (req, res) => {
  const request = await verifiedIdClient.createPresentationRequest({
    acceptedIssuers: [CLOUDSTRUCC_ISSUER_DID],
    credentialType: 'CloudstruccEmployeeCredential',
    requestedClaims: ['employeeId', 'email', 'employmentStatus', 'assuranceLevel'],
    callbackUrl: `${PUBLIC_BASE_URL}/api/verifier/callback`,
    state: crypto.randomUUID()
  });

  res.json({ qrCodeUrl: request.url, transactionId: request.id });
});

8. Aries / ACA-Py Lab

The Aries lab is useful for Bifold/Credo wallet testing, DIDComm, mediator-based mobile flows, AnonCreds schema/credential-definition publishing, and non-Microsoft interoperability.

When a DID service is necessary

  • You need DIDs outside Microsoft Verified ID.
  • You need DIDComm endpoints and mediator services.
  • You need to publish AnonCreds schemas/credential definitions.
  • You need a wallet ecosystem such as Bifold/Credo.
services:
  acapy-mediator:
    image: ghcr.io/openwallet-foundation/acapy-agent:py3.12-1.6.0
    command: start --label "Cloudstrucc Mediator" --admin 0.0.0.0 3001 --admin-insecure-mode --mediator-invitation --open-mediation
    ports: ["3000:3000", "3001:3001"]

  acapy-issuer:

    command: start --label "Cloudstrucc Issuer" --admin 0.0.0.0 4001 --admin-insecure-mode --auto-provision
    ports: ["4000:4000", "4001:4001"]

  acapy-verifier:
    image: ghcr.io/openwallet-foundation/acapy-agent:py3.12-1.6.0

Do not expose ACA-Py admin APIs with --admin-insecure-mode outside local development.

9. OIDC4VC Guidance

OIDC4VC makes sense when Cloudstrucc needs OpenID/OAuth-style wallet interactions beyond a Microsoft-only flow. Use OID4VCI for credential issuance and OID4VP for presentation. Keep Aries DIDComm, Microsoft Verified ID, and OIDC4VC adapters separate until compatibility is proven through wallet testing.

10. Security Controls

  • Require phishing-resistant MFA for issuer administration.
  • Require YubiKey/passkey authentication for employee credential issuance.
  • Validate callback state and correlation IDs.
  • Use HTTPS everywhere.
  • Do not log raw credentials, tokens, private keys, or unnecessary personal data.
  • Separate sandbox, staging, and production issuers.
  • Keep VON Network restricted to local development.
